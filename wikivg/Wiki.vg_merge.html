Wiki.vg merge

The following discussion is closed. Please do not modify it.

wiki.vg content is now being imported and integrated into this wiki. For further information and discussion, please refer to MCW:Projects/wiki.vg merge and the "wiki.vg merge" thread in Discord #project-forum. --GIM Dianliang233 (talk) 02:35, 29 November 2024 (UTC)

wiki.vg, a wiki that documents Minecraft's protocol extensively, is sunsetting on November 30, 2024. As the announcement posts reads, "The final content will be archived and made available for download in MediaWiki's XML export format under the CC-BY-SA license, along with an archive of images and other media." The sunsetting announcement can be read here.

The creator of wiki.vg personally hopes that wiki.vg's content will be merged into this wiki, and this sentiment is shared by some wiki.vg users. So we must answer two questions:

  1. Should we merge wiki.vg's content into minecraft.wiki?
  2. If yes, how exactly should we go about doing that? A notable trouble is that wiki.vg and minecraft.wiki's licenses are not the same, they are CC-BY-SA and CC-BY-NC-SA respectively. SA (ShareAlike) dictates that the content must be released under the same license. According to discussion on Discord there are (seemingly simple) workarounds to this, but I'll let those users explain it clearly in response.

I'd also like to note that the board will discuss this topic in a future board meeting. The current planned board meeting (November 16, 19:30 UTC) is looking to be very long (2+ hours), so this topic might be postponed to a future meeting, we're not 100% sure. - Harristic / Talk 02:26, 13 November 2024 (UTC)

I personally  support this merge. Wiki.vg is an important resource and of course should have a new home, and since its content is something we likely should be documenting anyway (as far as I can tell), a merge can't hurt. - Harristic / Talk 02:26, 13 November 2024 (UTC)

 Support (I'm primarily a wiki.vg contributor.) Some thoughts:
  • It is probably not worth trying to import the very outdated client and server tutorial pages. Updated information on the mechanics documented in the client tutorial would, I think, be better integrated into the protocol documentation. There are also some other pages that are outdated and/or have equivalent or better substitutes on Minecraft Wiki.
  • The protocol documentation is split across many pages, which have simple names like "Chunk Data", with no subpage prefix. The pages listed under Current Protocol Specification on the main page should probably be made subpages of the main Java Edition Protocol page.
  • Some frequently used templates - Type, Change, and possibly others - have very generic names, and would be made somewhat awkward by a lengthy qualifying prefix.
  • There are currently a number of problems around management of protocol versions, resulting from shortcomings of MediaWiki:
    • The current process of documenting new versions is rather tedious, involving manual construction of diffs followed by manual merger into the main article.
    • Documentation for old protocol versions is available only via past revisions of the Protocol article. As such, mistakes left in old versions can never be corrected. Another problem is that links and transclusions always refer to the latest version, which can be confusing.
    • The usual approach of documenting past versions on a separate "History" section of the article is not viable here due to the sheer volume of changes taking place.
    • The ideal solution to this in my mind would function like git, with separate branches for each version. A reasonable approximation might be to have separate pages for each version (subpages included). These could be split from the pre-release documentation when new versions come out, after merging any relevant changes made to the previous release's page in the meantime. (People with more experience can make the call on whether this would be too unwieldy to manage.)
LassiPulkkinen (talk) 07:23, 13 November 2024 (UTC)
 Support as long as the protocol documentation resides in a separate namespace (forever). This is for a variety of reasons, some listed below:
  • The license issues mentioned above. Mixing licenses in one namespace does sound like a pain, and relicensing the content is likely not an option (because many contributors probably can no longer be contacted).
  • Protocol pages have a different audience. For instance, a reader of the protocol wiki is expected to have an inkling about what TCP/IP is, but a reader of the general Minecraft wiki is not.
  • Protocol pages have a different structure. They are not about specific things in the games (entities, mechanics, etc.) with infoboxes and trivia, they are usually tables (or lists of tables) more akin to a C structure definition than a wiki article. (For instance, https://wiki.vg/Inventory)
  • wiki.vg did not have a style guide, so quite a few articles will have to be modified to fit that of this wiki. There's not time for that before wiki.vg goes offline, so a separation will be useful for the early period. (This reason matters less because it will be fixed with time, but the other three will not.)
License issues and the logical divide in content seem to warrant a separate namespace. Some additional thoughts below:
  • What is the plan for https://wiki.vg/Main_Page#Tools_.26_Mods? Articles here do not fit Minecraft_Wiki:Notability, and some violate Minecraft_Wiki:Wiki_rules, as they mention cheats. I assume they will be axed according to the rules, which is disappointing because some of these tools are useful.
  • I'm excited by the prospect that some of the long-standing issues with wiki.vg could be fixed during this merge, namely the immutability of old protocol version pages.
(disclaimer: I am mostly a wiki.vg editor/protocol discord participant, but I believe this discussion pertains to me since it appears as though this will be the new home for Minecraft protocol documentation.) bigfoot (talk) 08:51, 13 November 2024 (UTC)
